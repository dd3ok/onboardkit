# Specify Guide

Use this guide for user-visible behavior, product changes, or tasks whose scope should be agreed before design.

## Inputs

- `brief.md` or user request
- Product context
- Constraints

## Outputs

- `specs/<slug>/spec.md`

## Steps

1. Define the problem and users.
2. State what must change and why.
3. Add acceptance criteria.
4. Add non-goals and constraints.
5. List edge cases.
6. Identify success metrics or review criteria.

## Completion criteria

- The spec explains what and why, not implementation details.
- Acceptance criteria can drive tasks and verification.
